## 3. 検索パラメータへの変換可能性

### 3-1. InterviewDetail → job-platform 検索パラメータ: ✅ 機械変換可能

| InterviewDetail フィールド | job-platform パラメータ | 変換方法 |
|---|---|---|
| `desired_job_type_1`（"営業 / 法人営業 / 新規開拓"） | 職種カテゴリ | "/"でsplit → 大カテゴリをマスタ照合 |
| `desired_prefecture`（"東京都"） | 都道府県 | そのまま |
| `desired_salary_min`（整数・万円） | 年収下限 | そのまま |
| `desired_industry_1`（"IT / ソフトウェア"） | 業種 | "/"でsplit → マスタ照合 |

**AI変換不要。** カテゴリマスタとのマッピングテーブル（職種大分類→job-platform職種コード等）を1つ作れば機械的に変換できる。コスト: マッピング作成の初期工数のみ。

### 3-2. タイプ診断（自由文）→ 検索パラメータ: ⚠️ AI変換が必要

タイプ診断の出力は自由文テキストだが、「検索条件（推奨）」セクションに職種・業種・年収・エリアが含まれる場合が多い。

**変換方法の候補**:
1. **リアルタイム AI 抽出**: タイプ診断テキストを Claude/Gemini に渡し、構造化 JSON（職種・エリア・年収等）に変換。精度は高い（元がAI出力で半構造的）がコスト: 1回あたり約$0.01〜0.03（Haiku/Flash）。
2. **バッチ抽出＋キャッシュ**: タイプ診断生成時に裏で構造化抽出し、専用カラム/テーブルに保存。1回限りのコスト。
3. **SKILL.md の出力ルール改修**: タイプ診断時に構造化 JSON も同時出力させ、専用フィールドに保存（最もクリーン。新規診断から自動適用、既存156件はバッチ変換）。

### 3-3. ランク付き提示（「本人希望にA〜B／Cで近い」）の成立可能性

**既存のブックマーク `aiMatchRating`（A/B/C/D）はそのまま使える。** ただしこれは「既にCAがブックマークした求人」への評価であり、「条件が近い求人」レーンで新規検索した求人に対しては新たにランク付けが必要。

**新規求人へのランク付け方法**:
- seed検索で候補求人を取得 → 候補者コンテキスト（InterviewDetail + タイプ診断テキスト）と求人情報を突き合わせ → AI で A/B/C/D 評価。これは既存の `analyze-batch` ルート（ブックマーク解析）と同じパターン。
- 1候補者あたりの検索seed生成は軽量（InterviewDetail の構造化データをパラメータ化するだけ）。ランク付けはブックマーク解析の延長で既存ロジックを流用可能。

## 4. seed差し替えの実装方針案（実装はしない）

### 推奨: InterviewDetail ベースの構造化seed（主軸）＋ タイプ診断由来の深層seed（補助）

**第1段階（InterviewDetail seed・コスト低・即実装可）**:
候補者の最新面談 InterviewDetail から `desired_job_type_1`、`desired_prefecture`、`desired_salary_min` を取得し、job-platform 検索 API のパラメータに変換して「条件が近い求人」を検索。マッピングテーブル（職種大分類→APIコード等）を portal 側に置く。キャッシュは候補者IDをキーに24時間TTL（InterviewDetail は面談ごとに更新されるため頻繁には変わらない）。対象: 1,081人（面談済みかつ条件入り）。

**第2段階（タイプ診断 seed・AI変換要・156人→増加中）**:
タイプ診断テキストから構造化条件を抽出する専用フィールド `advisor_type_diagnosis` (JSON) を Candidate または新テーブルに追加。SKILL.md の出力ルールに「構造化JSONも同時出力」を追記し、新規診断は自動保存。既存156件はバッチ変換。このデータを InterviewDetail seed に重ね合わせ（タイプ診断があればそちらを優先、なければ InterviewDetail フォールバック）。将来的にタイプ診断の充填率が上がれば主軸に昇格。

**変換の置き場所**: portal の API ルート `/api/external/candidate-site/recommendations` として新設。mypage BFF が候補者IDで呼び出し、portal 側で seed 生成→ job-platform 検索→ ランク付け→ 結果返却を一貫処理。キャッシュは portal 側（Redis or DB テーブル）に TTL 24h で保持。

## 5. 補足: advisor-context.ts の盲点

現在のAIアドバイザーのコンテキスト構築（`src/lib/advisor-context.ts`）は **InterviewDetail / InterviewRecord のデータを含んでいない**。面談で得られた希望条件・職歴・評価がAIアドバイザーに渡されていないため、タイプ診断の精度が書類・メモ・ブックマークのみに依存している。将来 InterviewDetail を context に含めれば、タイプ診断の精度向上が見込める（副次的な改善提案・今回のスコープ外）。
