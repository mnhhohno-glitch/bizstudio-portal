## 1. 保存構造

### テーブル・列

| 項目 | 値 |
|--|--|
| テーブル | `candidate_files`（Prisma: `CandidateFile`） |
| 列 | `ai_analysis_comment`（`String? @db.Text`） |
| 補助列 | `ai_match_rating`（A/B/C/D 総合判定）、`ai_analyzed_at`（分析日時） |

### 1つの文章か、分かれているか — **1つの文章（同一列に連結）**

「おすすめポイント（本人向け）」と「選考分析（CA向け）」は**同一の `ai_analysis_comment` 列に連結されている**。別々の列ではない。

典型的な構造:
```
【会社名】求人タイトル

■ 本人希望: B
■ 通過率: C
■ 総合: B

◆ おすすめポイント（本人向け）
（前向きな推薦コメント。3〜5文。）

◆ 選考分析（CA向け）
（必須要件充足状況、懸念点、通過戦略。CA専用。）
```

### 関連する別の列: `ca_comment`

`CandidateFile.caComment`（`ca_comment`）は**手動のCAコメント**であり、AI分析とは完全に独立。ACTIVE ブックマークで `caComment` が入っている行は**1件のみ**（ほぼ未使用）。

## 6. 実装の推奨

### 結論: **見出しベースの切り出しで進めてよい（フェイルクローズ前提）**

理由:

1. **ACTIVE ブックマークの 99.0% が安全に切り出し可能**。非表示は23件のみ（旧フォーマット/CFのみ）
2. **表記ゆれは実質ゼロ**。セクションヘッダーとして意味のあるバリエーションは標準形 + trailing `**` のみ。既存正規表現で全パターンをカバー
3. **順序は 100% 安定**。逆順は 0件
4. **切り出しロジックは既に本番稼働中**（`extractCandidateFacingComment()`）
5. **サンプル10件の目視で CA内容の混入は 0件**

### 安全策

1. **フェイルクローズ**: 両見出しが正順で存在しない場合は `null` を返す（何も表示しない）
2. **既存 `extractCandidateFacingComment()` のバグ修正を推奨**: CF-only 行（20件）で CA内容を含めて返している。CA見出しがない場合は `null` を返すべき（フェイルクローズ化）。sync-ca-comments にも影響するため要確認
3. **抽出はサーバーサイド（portal）で行う**: mypage に `aiAnalysisComment` の生テキストを渡さない。portal の favorites API で抽出済みテキストだけを返す

### 保存構造を分ける必要はあるか — **不要**

- 現在の単一列構造で、見出しベースの切り出しが 99%+ の精度で機能している
- プロンプトが「必ず2セクション両方を出力」「おすすめポイント→選考分析の順」を明示指定しているため、今後の新規生成でも構造は安定する
- 列を分けると、既存の5,713件のマイグレーション、CA画面の編集UX（全文テキストエリア）の改修が必要になり、コスト対効果が悪い
