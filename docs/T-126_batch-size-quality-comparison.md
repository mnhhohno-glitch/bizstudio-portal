# T-126 求人分析 バッチサイズ 5件 vs 10件 品質比較（検証のみ・2026-07-02）

対象: bizstudio-portal（staging 環境）。テスト求職者「大野 テスト」（candidateNumber 5999999）の
ブックマーク求人票 先頭10件を、**同一入力**で「5件/バッチ（2バッチ）」と「10件/バッチ（1バッチ）」で分析し比較。

⚠️ 本番の batch size は変更していない（batchSize はリクエストパラメータで、client 既定は 5 のまま）。
採否は将幸さんが本レポートを見て判断する。

## 実行条件
- モデル: Opus 4.6（`CLAUDE_MODEL_ANALYSIS`）、max_tokens=16000、temperature=0.7
- 5件/バッチ: セッション S5 の batch0（1〜5件目）+ batch1（6〜10件目）
- 10件/バッチ: セッション S3 の batch0（1〜10件目・単一コール）
- 出力フォーマット（3軸マーカー ■本人希望/■通過率/■総合、◆おすすめポイント、◆選考分析）は両者とも同一で、Phase2 でも不変。

## 結果サマリ

| 観点 | 5件/バッチ | 10件/バッチ | 所見 |
|---|---|---|---|
| 全求人の3軸マーカー出力 | 10/10 揃う | 10/10 揃う | どちらも欠落なし |
| 両セクション（本人向け◆/CA向け◆）具備 | 10/10 | 10/10 | どちらも完備 |
| 本人向け◆コメント平均長 | 287字 | 260字 | 10件は約 **-9%** 短い |
| CA向け◆コメント平均長 | 398字 | 383字 | 10件は約 **-4%** 短い |
| 3軸レーティング一致 | （基準） | 8社中4社完全一致、4社が±1ランク差 | 下記 |
| 1求人あたり出力/所要 | ~2バッチ×100s、cache write 1回 | 1バッチ 185s、cache write 1回 | 10件は API 往復・cache write が半減 |

## レーティング差（±1ランクの範囲）

| 会社 | 5件（希望/通過/総合） | 10件（希望/通過/総合） |
|---|---|---|
| パーソルクロステクノロジー | D/D/D | D/D/D（一致） |
| 楽天モバイル | D/D/D | D/D/D（一致） |
| 株式会社MTG | D/D/D | D/D/D（一致） |
| SBSホールディングス | D/D/D | D/D/D（一致） |
| マンパワーグループ | B/A/A | B/B/B |
| 南海辰村建設 | B/B/B | A/B/A |
| 株式会社ラック | C/D/D | B/C/B |
| M&Aキャピタルパートナーズ | D/D/D | C/D/D |
| 新日本溶業 | C/B/B | B/B/B |
| 鈴茂器工 | B/B/B | A/B/A |

※ 差はいずれも**隣接1ランク**。ただし temperature=0.7 のため、5件×2回の再実行でも同程度の揺れは出る
（＝この差の一部はバッチサイズではなくサンプリングゆらぎ由来）。傾向としては 10件の方が
やや楽観側（B→A 等）かつコメントが短めで、1求人あたりの注意配分が薄まる兆候はある。

## 判断材料（コスト面）
- 10件/バッチは 求人1件あたりの cache write・API往復・固定プレフィックス負担が概ね半減する
  （バッチ数が半分になるため）。Phase2 の「シングルバッチ run は cache_control 無し」と組み合わせると、
  10件×1バッチ = 単一バッチ扱いになり cache write すら発生しない（全入力を素の入力単価で1回）。
- 一方で、単一コールに10求人分の入力（各3000字上限）＋候補者context＋skill が乗るため、
  1コールの入力トークンは大きくなる（本検証の10件単一バッチ uncached ≈ 40K）。トークン総量では
  「5件×2バッチ（skill/context がキャッシュ再利用）」と大きくは変わらないが、コール数・レイテンシは減る。

## 結論・推奨
- **品質**: 10件でも致命的劣化はない（フォーマット完備・欠落なし）が、コメントが約1割短く、
  レーティングにやや楽観ドリフトの兆候。求人1件ごとの推薦文の厚みを重視するなら 5件が安全。
- **コスト/スループット**: 10件はコール数・キャッシュwrite・レイテンシを削減できる。
- **推奨**: 既定は **5件を維持**（コメント品質優先）。コスト最優先の局面や大量再分析時に限り
  10件を選べるようにする案は将幸さんの判断に委ねる。本タスクでは本番 batch size は変更しない。

## 付記
- 検証は staging（本番と同一 Postgres 共有）で大野テストのみ使用。本番求職者データは読み取りなし。
- 出力フォーマット仕様（3軸・2セクション・総合優先順位まとめ）は Phase2 で一切変更していない。
